HPA Scaling Latency Analysis

\n
\nAuthor: DevOps Team\n•\nDate: Oct 24, 2023\n•\nID: DOC-8821\n
\n

This document details the investigation into horizontal pod autoscaler delays observed during peak load events in the us-east-1 production cluster. The latency resulted in a 5% drop in request success rate.

\n

Observation

\n

During the daily traffic spike at 09:00 UTC, the `payment-service` HPA failed to scale replicas from 5 to 20 within the expected window (2 minutes). Instead, scaling took approximately 8 minutes.

\n
apiVersion: autoscaling/v2\nkind: HorizontalPodAutoscaler\nmetadata:\n  name: payment-service-hpa\nspec:\n  scaleTargetRef:\n    apiVersion: apps/v1\n    kind: Deployment\n    name: payment-service\n  minReplicas: 5\n  maxReplicas: 50\n  metrics:\n  - type: Resource\n    resource:\n      name: cpu\n      target:\n        type: Utilization\n        averageUtilization: 70
\n

Root Cause Analysis

\n

Analysis of the metrics server logs indicated a timeout when scraping custom metrics from the Prometheus adapter. This caused the HPA controller to operate on stale data, delaying the scale-up event.

\n
    \n
  • Metrics Server latency increased to >5s.
    • \n
  • Prometheus adapter was OOM killed twice during the incident window.
    • \n
  • Network policy restrictions initially suspected but ruled out.
    • \n
\n

Remediation

\n

We have increased the memory limits for the Prometheus adapter and tuned the scrape interval to reduce load.
